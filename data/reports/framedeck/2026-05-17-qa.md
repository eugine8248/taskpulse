**Critical:** 1 | **Important:** 6 | **Minor:** 9

Scope: read-only QA of v0.10 → v0.14 surface (schema drift, bundle health, endpoint coverage, error consistency, edge cases, notification correctness, mobile-readiness via code-read, multi-tab, Yjs, performance). v1.0-pivot work skipped per scope. Live API was running on `localhost:8080` during the audit.

Verification commands run:
- `npx prisma validate` — schema OK
- `prisma db pull --print` — DB matches `schema.prisma` (all v0.10–v0.14 models present with correct indexes)
- `tsc --noEmit` — clean for `api/` and `client/`
- `vite build --emptyOutDir false` — clean, **248.51 KB gzip** main bundle (vs v0.9 baseline ~210 KB → ~+18 % growth across four releases)
- `npm audit` — `api/` and `collab/` 0 findings; `client/` 1 moderate dev-only (esbuild GHSA-67mh-4wv8-2f99, transitive via vite)
- ~10 curl probes for endpoint existence + error shape

## 1. Schema drift

PASS overall. `prisma db pull --print` matches `schema.prisma` for every v0.10–v0.14 model (`GuestLink`, `CardComment`, `CardSubmission`, `SubmissionComment`, `BoardNotification`, `BoardNarration`, `TimeEntry`, `Subscription`). All declared indexes survived push (`@@index([cardId, createdAt])` on CardComment, the dual auth-tuple indexes on TimeEntry, etc.).

### MINOR — `Subscription` model exists despite v1.0-pivot work being "in flight"

The `Subscription` model is already in `schema.prisma` (lines 449–467) and in the live DB. The audit scope said the v1.0-pivot agent is mid-build, so this is informational — the schema change has shipped even though the surface is unstable. Existing users get `Free/active` rows lazily via `ensureSubscriptionRow` (`api/src/routes/auth.ts:49`, `billing.ts:65`).

### MINOR — `BoardNarration.expiresAt` has no index

`prisma/schema.prisma:413` adds `expiresAt: DateTime?` for narrations attached to expiring guest links. The public replay path checks `n.expiresAt && n.expiresAt.getTime() <= Date.now()` (`api/src/routes/narrations.ts:237`). Not currently filtered in any LIST query so no index hot-path. If a future cron sweeps expired narrations, it'll do a full scan.

## 3. API endpoint coverage

Spot-checked ~10 endpoints against V010–V014 reports via curl:

| Endpoint | Documented | Reality | Status |
|---|---|---|---|
| `GET /api/health` | implicit | `{ok:true, ts:...}` | OK (but shape NOT `{success, data}` — see §4) |

| `GET /api/cards/<cuid>/submissions` | V011 — auth-or-guest list | 404 `{success:false, error:"Card not found"}` for non-existent card; 401 for existing card w/o auth | OK |

| `POST /api/cards/<bad>/time/start` | V014 — auth-or-guest | 404 `{success:false, error:"Card not found"}` (no auth supplied) | OK shape; **see §4 — auth ordering inconsistency** |

### IMPORTANT — auth-or-guest endpoints do `findCard` BEFORE checking auth, leaking card existence

Several v0.10–v0.14 routers structure resolveCardAndAccess as: SAFE_ID → findCard → if missing, 404 → THEN try guest/JWT. This means an unauthenticated caller can distinguish "card cuid does not exist" (404 `Card not found`) from "card cuid exists but no auth" (401 `Auth required`). Files affected: `api/src/routes/submissions.ts:94-141`, `submissionComments.ts:83-95`, `timeEntries.ts:104-112`, `guest.ts:430-440`.

The v0.14 `boardTimeRouter.get('/:boardId/time/summary')` (`timeEntries.ts:487-500`) gets it right — auth FIRST, then board lookup. The other routers were written before this pattern was identified.

**Impact:** Allows existence-confirmation of known/leaked card IDs without auth. Cuid entropy makes brute force infeasible; a card ID that already leaked elsewhere can be confirmed here.

## 4. Error response consistency

Mostly conformant. Errors mostly use `{success:false, error:<string>}`. Status codes mostly correct (401 vs 403 vs 404 vs 413 vs 429 all observed in code).

### IMPORTANT — `/api/health` does NOT use the `{success, data}` envelope

`api/src/index.ts:111` returns `res.json({ ok: true, ts: Date.now() })`. Every other endpoint uses `{success, data}` or `{success, error}`. Minor footgun for a future "is the API alive?" client that parses by envelope.

### IMPORTANT — error string drift between equivalent auth failures

For "I need a Bearer token and you didn't supply one":
- `notificationsRouter` (via `authMiddleware`) → 401 `"No token"`
- `boardTimeRouter` (manual auth) → 401 `"Auth required"`
- `submissionsRouter` (manual auth) → 401 `"Auth required"`
- `submissionCommentsRouter` (manual auth) → 401 `"Auth required"`
- `timeEntriesRouter` (manual auth) → 401 `"Auth required"`

For "Bearer present but JWT verify failed":
- `authMiddleware` → 401 `"Invalid token"`
- manual paths → 401 `"Invalid token"`

A client doing `if (err === 'No token')` will silently break on routes that use the manual path.

**Recommended fix:** Pick one. The `authMiddleware` string should be authoritative.

### IMPORTANT — Stripe webhook responses use plain text, not `{success, error}` (`api/src/routes/billing.ts:228, 233, 243, 272`)

This is technically correct — Stripe expects plain-text responses. But it diverges from the rest of the API. A monitoring system grepping for `"success":false` in 5xx responses will miss webhook handler failures.

**Recommended fix:** Acceptable as-is; document the divergence.

### MINOR — `/api/comments/:id` (legacy) DELETE returns `{success: true}` with no `data` (`api/src/routes/comments.ts:36`)

All other endpoints return `{success: true, data: ...}`. Cosmetic.

### MINOR — `feedbackByUserId` not exposed in `shapeSubmission` output (`api/src/routes/submissions.ts:182-204`)

`feedbackByUserId` is on the row but stripped by the shape helper. The V011 report says "feedbackVerdict | feedbackNote | feedbackAt" are surfaced, doesn't mention feedbackByUserId. If a client wants to render "approved by Alice" it can't — has to join via Yjs/board state. Intentional minimization but worth flagging.

## 5. Edge cases

### CRITICAL — concurrent guest-token timer starts may bypass the "≤1 running per author" invariant on Postgres

`api/src/routes/timeEntries.ts:256-287`. The `$transaction` correctly auto-stops prior running entries and creates the new one. Under SQLite (current DB), the DB-level write lock serializes the two transactions and the invariant holds. Under Postgres (the v1.0-pivot doc mentions Postgres readiness) with default `ReadCommitted` isolation, two near-simultaneous starts can BOTH find no running entries and BOTH create new rows.

**Repro intent (not exercised — read-only audit):** Two browser tabs of the same guest token POST `/start` within ~10 ms of each other.

**Impact:** Today (SQLite) — none. After Postgres migration — analytics double-counting, "stop my running timer" arbitrarily picks one and orphans the other.

**Recommended fix:** Add `{ isolationLevel: 'Serializable' }` to the `$transaction` options before the migration, OR add a partial unique constraint at the DB level (Postgres-only):
```sql
CREATE UNIQUE INDEX one_running_user ON "TimeEntry" ("authorKind", "authorUserId") WHERE "endedAt" IS NULL;
CREATE UNIQUE INDEX one_running_guest ON "TimeEntry" ("authorKind", "authorGuestToken") WHERE "endedAt" IS NULL;
```

### IMPORTANT — "guest's parent board is deleted while polling" — guest sees confusing 404 mid-session

`api/src/routes/guest.ts:189` returns 404 if the board is gone. The client at `client/src/routes/GuestPage.tsx:78-87` does a one-shot fetch; if the user is already on the page when the owner deletes the board, the next interaction (comment POST, notification poll) returns 404 with `"Link not found or expired"` (because `resolveGuestToken` does `findUnique({where:{token}})` which returns the link even after board cascade — actually, `GuestLink.boardId` has `onDelete: Cascade` so the row IS deleted with the board, so token is missing → 404 `"Link not found"`). Either way the guest sees a generic "Link not found or expired" which masks the cause. Not a security issue but bad UX for the freelancer.

**Recommended fix:** Distinguish "link revoked", "link expired", "board deleted", "you got 404 for a different reason" in the response. Currently all collapse to one string.

### IMPORTANT — "user signs out mid-narration recording" — uploaded audio orphaned + no error surfaced

`client/src/routes/BoardPage.tsx:62-72` holds focus path in a ref + recorder state. If the user signs out mid-recording (`localStorage.framedeck.token` cleared by a different tab), the next `/api/uploads/audio` POST fails 401 silently. The recorder has no recovery path; the user clicks Save → silent failure → recording lost. No row is created so no DB orphan. If the audio uploaded BEFORE the sign-out and the row-create POST is what 401s, the audio file is orphaned on disk/S3 with no DB row pointing at it. There is no orphan-sweep for audio.

**Recommended fix:** Surface auth failure with a "you were signed out" toast; delete the orphaned key on row-create failure (best-effort cleanup mirror to the pattern in `cards.ts`).

### IMPORTANT — "submission's parent card is deleted" — submission rows cascade but in-flight feedback POSTs 404 silently

`Card` → `CardSubmission` has `onDelete: Cascade`. So deleting a card kills its submissions atomically. Good. But if the owner is in the middle of leaving feedback (the dialog is open in their browser), the POST returns 404 `"Submission not found"`. Client side, the feedback dialog has no graceful "card was deleted" recovery; the owner sees a generic error.

**Recommended fix:** Distinguish "submission not found" vs "submission withdrawn" vs "parent card gone" in the API for the client's benefit.

### MINOR — narration `viewCount` increments on revoked/expired narrations (`api/src/routes/narrations.ts:263-275`)

The view-bump endpoint catches the row-missing case but does NOT check `removedAt` or `expiresAt`. A bot could POST `/view` against a known-but-removed narration id forever. View count drifts up after the narration is unreachable. Cosmetic but breaks analytics integrity.

### MINOR — guest banner notification ack list grows unbounded in localStorage (`client/src/routes/GuestPage.tsx:103-119`)

The localStorage key `framedeck.guest.notifAcked.<token>` is a JSON array of notification IDs. A long-lived guest accumulates the array forever (no compaction). Each id is a cuid (25 chars), so 10000 ack'd notifications ≈ 250 KB — well under quota but worth noting.

## 6. Notification correctness

### MINOR — self-action skip relies on `actorUserId` being non-null (`api/src/lib/notifications.ts:57-62`)

The owner-and-editors fan-out skips a recipient if `recipient.userId === opts.actorUserId`. When the actor is a GUEST, `actorUserId` is `null/undefined`, so the comparison never matches and the actor (a guest) cannot be in the recipient set anyway. Works for guests. For authed actors it relies on the caller passing `actorUserId`. Spot-checked all `fireNotification` calls — every one passes either `actorUserId: who.mode === 'user' ? who.userId : null` or implicit `undefined` (which works the same). Subtle: passing `undefined` would let editor user `0` be self-notified if such a user existed. Minor.

### MINOR — owner-of-submission cross-board notification edge case (`api/src/routes/submissionComments.ts:257-278`)

When an authed user comments on a submission, the helper fires `toBoardEditors: true` with the cross-recipient comment that "user X commented on your work." The helper de-dupes by board-editor set. If the SUBMITTER is NOT a board editor (e.g. a Free-tier user submitting to a board they're not a collaborator on — this is hypothetical, the auth gate at line 138 ensures only collaborators or guests can submit), the submitter would not get the notification. Today the access gate prevents this case from arising, but a future change widening submission access (e.g. "anyone-with-link can submit") would silently drop notifications to the submitter.

### MINOR — `time_logged` kind is fired even when actor is the owner via guest path (`api/src/routes/timeEntries.ts:684`)

The skip-self check is `if (actorUserId && board.ownerId === actorUserId) return;`. A guest contributor logging time on the owner's board → fires notification → owner correctly gets pinged. But if the owner of a board ALSO has a separate guest link to their own board (unusual but possible — e.g. testing the guest UX), the owner stops the timer as a guest → `actorUserId` is null → skip-self does NOT trigger → owner gets a notification about themselves. Edge case.

## 7. Mobile readiness (code-read only)

### IMPORTANT — `VideoReviewPlayer` scrub bar markers are not touch-friendly (`client/src/components/board/VideoReviewPlayer.tsx`)

Not opened in this audit, but the v0.13 design ships a video player with frame-anchored comment markers on the scrub bar. Touch precision on a scrub bar with markers at 200ms granularity is poor on a 4-inch screen. No mobile-specific layout flagged in the routing of BoardPage. Concern based on UI-pattern read; not verified by running.

### MINOR — `VoiceRecorder` MediaRecorder API needs a browser permission

`client/src/components/board/VoiceRecorder.tsx` uses MediaRecorder. On iOS Safari, the API was unsupported until iOS 14.3 and is still flaky on permissions. The component should surface "your browser doesn't support audio recording" but I did not verify the fallback path.

### MINOR — guest page lacks viewport/touch-zoom controls beyond ReactFlow defaults

`client/src/routes/GuestPage.tsx:46` uses default ReactFlow controls. ReactFlow's pan+zoom is touch-aware but the card detail panel slides out from the right edge — on a 375px phone width that's most of the screen and the canvas is unusable. Mobile guest flow likely degrades to "video review only" which may be acceptable for the v0.10 wedge.

## 8. Multi-tab behavior

### MINOR — two tabs of the same authed user starting timers race-correctly under SQLite

Per §5 (same finding). Under current SQLite deployment, two tabs starting a timer on different cards within the same user will serialize correctly via the `$transaction` write lock — first tab's `start` auto-stops nothing (no prior), second tab's `start` auto-stops the first tab's entry. The first tab's UI shows a stale "running" until it refetches; no polling mechanism is visible in the client code.

**Recommended fix:** The TimeTrackingSection could poll `GET /api/cards/:cardId/time` to detect when "my" running timer was stopped by another tab.

### MINOR — Yjs sync across two tabs is fine but BoardPage's local React state holds the photo URL independently of Yjs

`api/src/routes/cards.ts` updates `photoUrl/photoKey` server-side. Yjs only syncs CARD POSITIONS (`Y.Map<{x,y}>`). Photo changes propagate via the next full `/api/boards/:id` refetch, NOT via Yjs. Two tabs editing the same card photo simultaneously will see whatever wins the API race.

### MINOR — notification bell does not push; both tabs poll independently every 30s

`client/src/components/NotificationBell.tsx` polls `/api/notifications/unread-count` (per V011 spec). Two tabs = two polls. Acceptable for v0.11 design; flag if scaling becomes a concern.

## 10. Performance

### IMPORTANT — unbounded list endpoints with no pagination

Several v0.10–v0.14 GET endpoints return ALL rows for a parent resource, no pagination:

- `GET /api/cards/:cardId/comments` (`guest.ts:472`)
- `GET /api/cards/:cardId/submissions` (`submissions.ts:425`) — filters `withdrawnAt: null` so soft-deletes don't pile up, but a busy card with 500 revisions still ships all of them
- `GET /api/submissions/:id/comments` (`submissionComments.ts:179`)
- `GET /api/cards/:cardId/time` (`timeEntries.ts:354`)
- `GET /api/boards/:boardId/narrations` (`narrations.ts:169`)
- `GET /api/guest/:token/cards/:cardId/comments` (`guest.ts:222`)
- `GET /api/boards/:id/guest-links` (`guest.ts:153`)

Per-card row count is bounded by user behavior, not enforced. A board with active video review on a single card easily lands 100+ `SubmissionComment` rows; a multi-week project might land 50+ `CardSubmission` revisions. Today none of these have an index that would suffer (`@@index([cardId, createdAt])` is fine for the small N) — the concern is payload size sent over the wire on every CardDetailPanel open.

**Recommended fix:** Add `limit`/`offset` query params (default 50, max 200), mirroring the notifications router pattern (`api/src/routes/notifications.ts:21-26`).

### IMPORTANT — `/api/boards/:boardId/time/summary` does multi-query aggregation in app code, not in SQL

`api/src/routes/timeEntries.ts:525-570`: pulls all cards, all entries for those cards, all submissions, all comments, then aggregates in JS. For a board with 200 cards × 50 entries/card = 10000 row roundtrip per dashboard load. Plus a separate `count` for comments resolved/total. Plus a separate findMany for `subs`.

Today's small dataset is fine; at scale this is the obvious N+1-class hotspot. SQL `GROUP BY authorKind, authorUserId, authorGuestToken SUM(durationMs)` would push the work to the DB.

**Recommended fix:** Replace with `prisma.timeEntry.groupBy(...)` and SQL `count` aggregates.

### MINOR — `resolveCardAndAccess` does `include: { board: { include: { collaborators: true } } }` on every auth-or-guest call (`api/src/routes/submissions.ts:94`, equivalents)

For each comment-post / submission-post / time-entry-start, the router pulls the full collaborator list of the board to find the actor's role. Acceptable for the current scale; not cached. With 50 collaborators × 10 endpoint hits/sec = manageable. Worth a memo for v1.0 prod.
