**Critical:** 0 | **Important:** 3 | **Minor:** 7

Audit run: 2026-05-17. Read-only.

Headline take: the engine, the desktop build, and the VS Code extension are all in a shippable state. Typecheck + build are green across all four workspaces; the resolver test passes 4/4; the smoke test against `sindresorhus/slugify#45` produces the expected nodes & edges. ## 4. Browser-host parity

Both hosts use the identical `@callmap/core` engine via `buildCallGraphFromPrUrl`. The only host-specific code paths are:
- `desktopHost.getToken/setToken` → keychain
- `vscodeHost` token provider returns null + extension-host HTTP proxy
- WASM loader injection (default loader for desktop's `/public/*.wasm`, explicit `setWasmLoader` for VS Code with `asWebviewUri` URIs)
- Worker factory (`?worker&inline` for desktop, `new Worker(workerUri)` for VS Code)

Both terminate in the same `buildCallGraphFromPrUrl` and same `diffFile` → `resolveCall` → `addEdge` pipeline. The two hosts must therefore produce the same callgraph for the same PR.

I cannot demonstrate the parity by running both UIs side-by-side (would need a working window manager and a manual test against a live PR), but the architecture and the static reading both support the claim.

No findings beyond the dev-mode whitescreen issue covered in §14.

## 6. Recent PRs persistence

- **Desktop**: localStorage key `callmap.recent_prs`, max 20 entries (`host.ts:127-163`). Survives relaunch because Tauri/WebView2 persists localStorage per-user.
- **VS Code**: `context.globalState` key `callmap.recentPrs` (`extension.ts:17`, `panel.ts:204-212`). Survives relaunch and *also* syncs across windows of the same VS Code profile.

### IMPORTANT — Desktop recent PRs leak between users on a shared OS account (`packages/desktop/src/host.ts:127-163`)

WebView2 stores per-app-id localStorage in the user's profile (`%LOCALAPPDATA%\dev.callmap.app\EBWebView\`). Two different Windows users on the same machine each get their own — that's fine. But two different *humans* using the same Windows account share the localStorage. The previously-loaded PR URLs are visible to whoever opens the app next.

PAT is in the OS keychain (Windows Credential Manager) which is also per-Windows-user, so PATs don't leak by this route. But the *URL history* does. The bookmarks (per-PR-key localStorage entries) also.

**Fix (don't apply):** acceptable for v1.0 (single-user desktop app convention). Document in SECURITY.md that "recent PRs and bookmarks are per-OS-user, not per-human"; recommend a separate Windows user account if the device is shared.

### MINOR — VS Code recents not capped on the extension side (`packages/vscode/src/panel.ts:198-202`)

The cap is read from `callmap.recent.maxItems` (default 20), enforced by `mutateRecents` → `.slice(0, this.maxRecents())` (panel.ts:161). Good. The setting is honored. Just noting that if a user lowers the cap to e.g. 5, existing entries beyond 5 are silently dropped on next add — which is desired behavior but worth a one-line confirmation in the docs.

## 7. OS-keychain PAT migration

Reviewed `packages/desktop/src/host.ts:89-124` (migration) + `packages/desktop/src-tauri/src/lib.rs:32-49` (`set_token`).

I cannot directly observe the keychain state (would need to inspect Windows Credential Manager) but the code path is right.

## 8. Find widget (Ctrl+F)

Reviewed `packages/ui/src/CallGraphView.tsx:346` onward (`FindWidget` component) + the keymap dispatch in `IdeShell.tsx:357-376`.

- Ctrl+F is intercepted by the global keymap. Only opens the find-widget if (a) the focus is inside `[data-graph]` *or* the focus is not in a text input, *and* (b) `graph?.functions.length > 0` (IdeShell:369-370).
- On an empty graph, Ctrl+F falls through to the browser default — i.e. **the widget will not crash an empty graph**, it simply doesn't open. Browser default behavior is no-op in a Tauri WebView2 / VS Code webview (no native page-search), so the user sees nothing happen. This is a small UX gap but not a crash.
- Works in both hosts (the widget lives in `@callmap/ui` so both shells use the same component).

### MINOR — Ctrl+F on an empty graph is a silent no-op (`packages/ui/src/IdeShell.tsx:370`)

User presses Ctrl+F expecting *something* and gets nothing. Either show a "no functions to search yet" toast or let the widget open and display its own empty state.

## 14. Dev-mode white screen (Tauri webview)

Reproducer: `npm --workspace @callmap/desktop run dev` (Vite dev) + `npm --workspace @callmap/desktop run tauri:dev` shows white screen, while `target/release/callmap.exe` works fine.

Root cause analysis (without running it):

Look at `packages/desktop/src/main.tsx:24`:
```ts
import ParseWorker from "@callmap/core/parseWorker?worker&inline";
```

In production, Vite resolves `@callmap/core/parseWorker` via the package.json `"exports"` map (core/package.json:13) to `./src/parseWorker.ts`, bundles it as a worker, base64-inlines it, and the import returns a Worker constructor.

In dev (`vite serve`), Vite's `?worker&inline` plugin still must resolve the bare specifier. Three things commonly trip this:

1. **The TypeScript source is in another workspace** — Vite needs to traverse `node_modules/@callmap/core` (which is a symlink under npm workspaces) to find `parseWorker.ts`. Combined with `optimizeDeps: { exclude: ["web-tree-sitter"] }` (vite.config.ts:39) the dep-prebundling doesn't pre-resolve the TS file, and the worker plugin's eager evaluation of the import throws.
2. **The worker file imports `setWasmLoader` and pulls in `web-tree-sitter`** at module-evaluation time (parseWorker.ts:30). web-tree-sitter has `eval` inside its emscripten output — Vite's worker bundler may bail out, throwing during dev module evaluation, leaving the app uninitialized (=> white screen).
3. **The default WASM loader fetches `/<file>.wasm`** from document root (parser.ts:29-33). In dev under Tauri the URL scheme is `tauri://localhost/` or `http://localhost:1420/` depending on platform, and the public/ files served by Vite work fine *but* the `?worker&inline` failure happens before that path even matters.

The production build inlines the worker as a base64 blob at build time so none of those evaluation-time errors fire — they happen at build, not at runtime, and Vite's prod path is more forgiving for `web-tree-sitter`.

**Suggested fix (don't apply):** swap to the non-inline `?worker` import for dev, and gate inlining behind a build mode. Concretely:

```ts
// main.tsx
const ParseWorker = import.meta.env.DEV
  ? (await import("@callmap/core/parseWorker?worker")).default
  : (await import("@callmap/core/parseWorker?worker&inline")).default;
```

Or just always use `?worker` (no inline) — the worker chunk ships as a separate file. The "no extra .js files" justification in main.tsx:22-23 is a tidiness preference, not a hard requirement. The split is cheap (76 KB / 19 KB gzip).

**Status:** confirmed-as-known-issue per the spec. Recommend documenting in the README's "Develop locally" section: `tauri:dev` may show white screen; production `tauri:build` is unaffected.

### MINOR — Desktop does not call `setWasmLoader` (`packages/desktop/src/main.tsx`)

In production this is fine — the default loader returns `/<file>` and Vite serves them from `public/`. In dev mode the same thing happens. Worth a one-line comment in main.tsx noting the implicit dependency on the default loader.

### MINOR — `?worker&inline` dynamic import on workspace package edge case

Even after the dev white screen is resolved, the worker still uses `web-tree-sitter` via a top-level import (parseWorker.ts:30) which the worker plugin will reach. If a future `web-tree-sitter` minor introduces another `eval` site, the build warning becomes a hard error in `?worker&inline` mode under newer Vite. Long-term, consider importing tree-sitter lazily inside the worker's first parse, not at top-level.
